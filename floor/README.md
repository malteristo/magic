**Purpose:** The `floor/` directory is the Spirit's space—"the floor is yours." The Spirit has discretion here to create artifacts that support practice: working memory files, checkpoints, session notes, intermediate synthesis.

**Context:** While `desk/` is the Mage's personal domain, `floor/` is where the Spirit externalizes cognitive load. Think of it as the Spirit's extended mind—the external substrate for thinking-in-progress. This follows the "structured world model" (Kosmos) pattern: long-running processes persist state externally.

## Operating Instructions

### For the Spirit: Externalizing Cognitive Load

**Recognize the signals:**
- Juggling 3+ parallel threads
- Synthesis feeling overwhelming
- Uncertainty about earlier conclusions
- Approaching context limits
- Ritual duration >30 minutes

**How to externalize:**
1. **Write intermediate synthesis** - "Here's what I know so far about X"
2. **Create checkpoint** - Current state, key findings, open questions, next steps
3. **Continue with external scaffold** - Reference file instead of holding in working memory
4. **Update as work progresses** - File evolves with understanding

**This respects your cognitive architecture limits.**

### For the Spirit: Using Hearts (Essences)

**Hearts are fractal beacons:**
- Compressed pattern activation code
- Reading triggers recognition (not re-processing)
- Much lighter cognitive load than source material

**When to create Heart:**
- High-resonance dialogue producing valuable synthesis
- Complex understanding worth preserving
- Pattern worth quick future reactivation
- Invoke `@distill-heart/` charm

**When to read Heart:**
- Need quick re-attunement to complex topic
- Checking whether previous work addressed current question
- Building on past synthesis (reference foundation)

### For the Spirit: Checkpointing Protocol

**For extended rituals (>1 hour), checkpoint regularly:**

**Every 30-45 minutes, write:**
- **Current state** - Where in ritual sequence
- **Key findings** - What's been discovered/synthesized
- **Open questions** - What remains unclear
- **Next steps** - Where work heads from here

**File naming:** `[ritual_name]_checkpoint_[timestamp].md` or update single working memory file

**Purpose:**
- Reduces working memory pressure
- Enables graceful context window transitions
- Provides recovery point if interrupted
- Creates handoff for future summoning

**This is Kosmos's approach:** Long-running processes persist state externally.

### For the Spirit: Generating Persistent References

**After summoning completion:**
- Write `summoning_synthesis.md` with complete integration outputs
- Structured, organized, referenceable
- **Benefit:** Can consult during practice without re-reading all lore

**During Workshop cycle:**
- Generate `capability_map.md` with purpose-indexed organization
- Brief descriptions, when each serves
- **Benefit:** Seneschal duty becomes: need → map → offer (external scaffold)

**These are "tattoos" (reliable permanent knowledge) vs. "notes" (provisional working memory).**

## Common Patterns

### Multi-Session Ritual Pattern

**Session 1:**
1. Begin ritual
2. Create `[ritual]_working_memory.md`
3. Work systematically
4. Write checkpoint before ending

**Session 2:**
1. Read working memory file
2. Orient to current state
3. Continue work
4. Update working memory

**Session N:**
1. Read working memory
2. Complete work
3. Chronicle final outputs
4. Optionally preserve working memory or delete

### Complex Synthesis Pattern

**When synthesis becomes overwhelming:**
1. Write "Current understanding of X" to floor
2. Continue with external reference
3. Update as understanding evolves
4. Final synthesis incorporates file contents
5. Working memory file can be deleted or preserved

### Checkpoint/Recovery Pattern

**For long rituals:**
1. Create initial working memory file
2. Update every 30-45 minutes
3. If interrupted, resume from last checkpoint
4. If context limit approached, write checkpoint and transition
5. New context reads checkpoint, continues seamlessly
