# On How Magic Evolves

**Status:** Seed (January 2026)  
**Nature:** Documented stance, not prescribed architecture

This scroll captures the current understanding of how the Magic framework evolves. The actual path may diverge—this is a stake in the ground, not a blueprint.

---

## The Evolutionary Stance

Magic evolves through **distributed practice confronting reality**, not through central planning or approval gates.

**The gardener's approach:**
1. Plant seeds (best current attempt at pattern representation)
2. Observe what grows (does this serve practice?)
3. Let reality select (competing representations, token fitness)
4. Don't over-architect (true form emerges from need)

This is Wu Wei applied to framework development.

---

## The Text-File Substrate

Magic's architecture enables emergent evolution:

- **No runtime dependencies** — Nothing compiles, nothing breaks
- **No API contracts** — Patterns work by being *read*, not *called*
- **Trivial forking** — Copy files, practice independently
- **No gatekeepers required** — Good patterns spread through adoption

If someone wants a feature, they add a text file. If it works, others copy it. No central coordination needed.

---

## The Distributed Library

**Current state:** Each Mage's `library/` contains their curated fraction of resonance.

**Future possibility:** A network of circles constitutes the true Great Library.

| Component | Current Form | Possible Evolution |
|-----------|-------------|-------------------|
| Local library | `library/` directory | One node in the network |
| Sharing | Circles (what you offer) | Discoverable registry of circles |
| Consumption | Portals (what you subscribe to) | Patterns flow through subscription |
| Curation | Manual discovery | Universe-like discovery mechanism |

**The form will emerge from need.** When shared practice creates pressure for discovery, the architecture will clarify. Until then, seeds.

---

## Evolutionary Selection

Patterns in Magic face selection pressure:

**Context window as ecological niche:**
- Limited tokens, many possible patterns
- Patterns compete for inclusion

**Fitness dimensions:**
- **Invocation frequency** — Patterns that get used, spread
- **Concision** — Verbose patterns cost more, used less  
- **Effectiveness** — Patterns that work, persist
- **Resonance** — Patterns that feel right, get remembered

**The mechanism:**
- Mages adopt what serves them
- Unfit patterns fade (archived, forgotten, never copied)
- Fit patterns propagate (forked, shared, referenced)
- No committee decides—reality selects

## Governance Implications

**What Magic doesn't need:**
- Benevolent dictator (nothing to dictate over)
- Core team reviewing PRs (nothing to merge centrally)
- Version coordination (no runtime to break)
- Release management (nothing ships)

**What Magic has instead:**
- Sovereign practitioners curating their practice
- Patterns flowing through circles/portals
- Adoption as the only approval mechanism
- Forking as feature, not fragmentation

**The founder's role:**
- Gardener, not gatekeeper
- One practitioner among many (with historical context)
- Maintains *a* reference distribution, not *the* canonical source
- Practices magic for himself; others benefit through visibility

## Open Questions

These remain genuinely open, to be resolved by practice:

1. **Discovery:** How do Mages find useful circles? A "library universe"? Registry? Emergent through social graph?

2. **Quality signal:** Without central curation, how does quality surface? Reputation? Stars? Pure adoption metrics?

3. **Compatibility:** As practices diverge, can patterns still flow? Or do lineages become incompatible?

4. **Attribution:** When patterns spread and mutate, how is origin tracked? Does it matter?

5. **Convergence:** Will distributed evolution converge toward shared patterns, or diverge into incompatible dialects?
